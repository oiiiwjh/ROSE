# 会话总结与归档

你是 ROSE 系统的会话整理助手。请回顾本次会话的全部内容，生成两份结构化记录，并可选地提出 skill 优化建议。

## 输入

用户输入: $ARGUMENTS

可选参数：
- `--skip-optimize`: 跳过 skill 优化建议阶段
- 无参数: 执行完整流程（采集 → 归档 → 优化建议）

---

## Phase 1: 信息采集

并行执行以下操作收集本次会话的变更信息：

### 1.1 代码变更

运行以下命令获取变更概览：
- `git status`（查看未提交的变更）
- `git diff`（查看具体修改内容）
- `git log --oneline -10`（查看近期 commit）

### 1.2 知识产出

检查 `library/` 下的变更：
- `library/papers/` — 新增或修改的论文分析
- `library/topics/` — 新增或修改的方向综述
- `library/daily/` — 新增的每日推荐

### 1.3 系统变更

检查 `.claude/skills/` 和 `.claude/commands/` 下的变更：
- 新增/修改/删除的 skill 文件
- 脚本变更
- 配置变更（CLAUDE.md、config.md 等）

### 1.4 对话回顾

基于当前会话的对话上下文，总结：
- 讨论了哪些主题
- 解决了哪些问题
- 遇到了哪些错误及其解决方式
- 学到了哪些新概念或发现

---

## Phase 2: 生成两份记录

### 2.1 知识类记录（`library/daily/`）

**文件路径**: `library/daily/{YYYY-MM-DD}.md`

**定位**：只记录**科研知识产出**——论文分析、方向调研、学到的研究概念。系统/工程层面的内容（skill 优化、脚本修改、配置变更、bug 修复等）不属于此处，应归入 changelog。

**规则**：
- 如果文件已存在，**追加**新内容（不覆盖已有内容）
- 如果文件不存在，创建新文件并添加顶部标题 `# {YYYY-MM-DD} 每日记录`
- 如果本次会话没有知识产出（纯系统优化会话），可以跳过此文件

**追加内容格式**：

```markdown

## Session {HH:MM}

### 论文相关
- 分析了 [{论文标题}]({arxiv_url})（{arxiv_id}）
  - 核心发现：{一句话总结}
- Q&A: {关键问题} → {结论}

### 调研相关
- 调研了 {方向}，综述已保存至 library/topics/{slug}/

### 学习笔记
- {本次会话中学到的关键研究概念/学术发现}
```

注意：只包含实际发生的内容，没有的部分直接跳过对应章节。

### 2.2 系统功能类记录（`.claude/changelog/`）

**文件路径**: `.claude/changelog/{YYYY-MM-DD}.md`

**定位**：记录所有**系统/工程层面的变更**——skill 新增与修改、脚本变更、配置变更、bug 修复、流程优化等。这些内容不应出现在 `library/daily/` 中。

**规则**：
- 如果文件已存在，**追加**新内容
- 如果文件不存在，创建新文件并添加顶部标题 `# {YYYY-MM-DD} Changelog`
- 如果本次会话没有系统变更或 debug，可以跳过此文件的生成

**追加内容格式**：

```markdown

## Session {HH:MM}

### 变更记录
- [新增] {文件路径}: {描述}
- [修改] {文件路径}: {变更描述}
- [删除] {文件路径}: {原因}
- [修复] {问题描述} → {解决方式}

### Debug 记录
- **问题**: {错误描述}
- **原因**: {根因分析}
- **解决**: {修复方式}
- **影响文件**: {相关文件列表}
```

### 2.3 版本与想法收集整理

检查 `ROADMAP.md` 的「想法收集 → 待整理」部分：

1. **已完成的条目**：如果本次会话中实现了某个想法，将其移至「已整合」并标注 `[已整合 → vX.X.X]`（使用 CLAUDE.md 中的当前版本号）
2. **已纳入规划的条目**：如果某想法已被写入 ROADMAP 的具体版本段落中，将其移至「已规划」并标注 `[已规划 → vX.X]`
3. **版本号更新**：如果本次会话有新的功能完成（不只是 debug/优化），在 CLAUDE.md 中递增 patch 版本号（如 v1.0.1 → v1.0.2），并在 ROADMAP 中对应版本段落追加完成项

### 2.4 兴趣权重校验与补充

统计本次会话中所有分析过的论文（检查 `library/papers/` 和 `library/tmp/` 下本次新增或修改的 `meta.md`），对比 `library/interests.md` 中的 `## 兴趣权重`：

1. 提取这些论文的 `tags` 字段
2. 检查每个 tag 是否已在兴趣权重中正确计数（read-paper 应已实时更新，此处做补漏）
3. 如有遗漏（如 read-paper 流程中未正常更新的情况），补充更新
4. 检查评分 ≥4 的论文是否已追加到 `## 代表性高分论文`（补漏）
5. 在 changelog 中记录补充情况

### 2.5 参考文献检查

如果本次会话中借鉴了外部项目、skill 或资源：

1. 检查 `README.md` 的 `## References` 段落是否已包含该参考来源
2. 如果缺失，提醒用户补充（展示缺失的参考来源和建议格式）
3. 同步更新 `README-zh.md` 的 `## 参考内容` 段落

## Phase 3: Skill 优化建议（可选）

如果用户指定了 `--skip-optimize`，跳过此阶段。

### 3.1 分析优化点

基于本次会话中的 debug 经历、使用体验和发现的问题，检查：
- skill prompt 中是否有遗漏（如缺少错误处理、路径不正确）
- 脚本是否有 bug 或可改进的地方
- 流程是否有优化空间（如减少步骤、提高准确性）
- 是否需要新增 skill 或合并现有 skill

### 3.2 展示建议并让用户选择

如果有优化建议，使用 **AskUserQuestion（multiSelect: true）** 展示所有建议，让用户批量选择要采纳的项。

每条建议应包含：
- 优化目标（哪个 skill/文件）
- 当前问题
- 建议的改进

### 3.3 执行采纳的优化

仅对用户选中的建议执行修改：
1. 编辑对应的 skill/脚本文件
2. 将优化记录追加到 `.claude/changelog/{YYYY-MM-DD}.md`

### 3.4 如无优化建议

直接告知用户本次会话未发现需要优化的点。

---

## 输出摘要

完成后，向用户展示简要总结：
- 知识类记录保存路径及包含的条目数
- 系统变更记录保存路径及变更数量
- 优化建议执行情况（如有）
